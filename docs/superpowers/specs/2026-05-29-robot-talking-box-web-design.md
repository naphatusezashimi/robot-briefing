# Design Spec: หุ่นยนต์ "กล่องพูดได้" แนะนำข้อมูลวิทยาลัยเทคนิคท่าหลวง — เวอร์ชันเว็บ

**วันที่:** 2026-05-29
**สถานะ:** อนุมัติแล้ว
**ขอบเขต:** สร้าง "หน้า" (จอ) + "เสียง" ของหุ่น เป็นเว็บแอปที่รันบนคอมได้ทันที โดยต่อยอดจาก "สมอง" (AI + ฐานข้อมูล) ที่ทำเสร็จในเฟส 1 — ยังไม่ทำตัวกล่อง/ฮาร์ดแวร์จริง

---

## ภาพรวมและที่มา

BRIEF เดิมวางหุ่นไว้เป็นอุปกรณ์บน Raspberry Pi 5 ที่ฟังเสียงด้วย Google Cloud STT และพูดด้วย Google Cloud TTS ระหว่างทางมีคำถามว่า "ควรทำเป็นเว็บแทนหรือไม่"

ข้อสรุปจากการ brainstorm: **มันคือ "กล่องพูดได้" กายภาพตัวหนึ่ง** (คนเดินมาพูดถาม มันพูดตอบ) ไม่ใช่เว็บลิงก์ให้คนสแกนเปิดในมือถือตัวเอง แต่เราจะสร้าง **"หน้าจอ" ของกล่องด้วยเทคโนโลยีเว็บ** เพราะได้ข้อดี 3 อย่าง:

1. **เสียงไทยฟรี** — ใช้ Web Speech API ของเบราว์เซอร์ ฟัง/พูดภาษาไทยได้โดยไม่ต้องจ่าย Google Cloud (สอดคล้องกับที่เลือกใช้ Gemini ฟรีไว้แล้ว)
2. **ทดสอบบนคอมได้ทันที** ก่อนซื้ออุปกรณ์ ตรงกับแนวทาง "โหมดจำลองก่อน" ของโปรเจกต์
3. **ยกขึ้นกล่องจริงได้** — พอประกอบกล่อง + จอสัมผัส ก็เปิดหน้าเว็บเดิมเต็มจอ (kiosk) กลายเป็นหน้าหุ่น ไม่ต้องทำ UI ใหม่

**เป้าหมายเวอร์ชันนี้:** เปิดเว็บบนคอม → พูดหรือพิมพ์คำถามเรื่องวิทยาลัย → หุ่นพูดตอบกลับ + แสดงคำตอบบนจอ + ใบหน้าแสดงอารมณ์ตามสถานะ โดยตอบจากฐานข้อมูลเท่านั้น (ไม่ตอบมั่ว)

---

## การตัดสินใจหลัก (locked)

1. **กล่องกายภาพ** ไม่ใช่เว็บแทนหุ่น — "เว็บ" เป็นแค่เทคโนโลยีวาดหน้าจอข้างในกล่อง
2. **เน้นพูด** (พูดใส่ → พูดตอบ) เป็นหลัก **มีช่องพิมพ์เป็นตัวสำรอง** ในจอเดียวกัน เผื่อคนไม่อยากพูด
3. **2 จอ** — จอใบหน้า (แสดงอารมณ์) + จอเนื้อหา (โลโก้/แชท/คำตอบ) ออกแบบแบบ **"แยกจอ รวมสมอง"**: เป็น 2 หน้าเว็บแยกกัน แต่ดึงสถานะจากเซิร์ฟเวอร์ตัวเดียว → ทดสอบบนคอมเปิด 2 หน้าต่าง, ประกอบกล่องก็แยกขึ้น 2 จอจริงได้ทันที
4. **เสียงไทยฟรี** ผ่าน Web Speech API (เบราว์เซอร์)
5. **สมองเดิมไม่แตะ** — `ai.py` / `database.py` / `college_data.json` รวมถึงกฎกันตอบมั่ว ใช้ต่อทั้งหมด
6. **สแตกเรียบง่าย** — Flask + HTML/JS/CSS ล้วน ไม่ต้องลง Node/เฟรมเวิร์กหน้าบ้าน

## สถาปัตยกรรม (3 ชั้น)

```
[ เบราว์เซอร์ ]                        [ เซิร์ฟเวอร์ Python ]        [ สมองเดิม ]
┌─ หน้า "ใบหน้า"  /face ──poll state─►  web/server.py (Flask)        ai.py
│   วาดตา/ปาก ตามสถานะ                  • เสิร์ฟ 2 หน้าเว็บ           database.py
│   (จอแสดงผลล้วน)                      • POST /ask → เรียก ask_ai()  college_data.json
│                                       • GET /state แจกสถานะปัจจุบัน
└─ หน้า "เนื้อหา" /screen ──/ask, /state► • POST /state รับสถานะจากหน้าจอ
    ไมค์ + พิมพ์ + แชท + พูดตอบ           • เก็บสถานะปัจจุบัน (in-memory)
```

- **ใบหน้า** = จอแสดงผลล้วนๆ ไม่ยุ่งกับเสียง/สมอง คอย poll `/state` แล้ว animate
- **เนื้อหา** = ตัวคุมงาน: ฟังเสียง (Web Speech STT) → ส่งคำถามไป `/ask` → ได้คำตอบ → โชว์บับเบิล + พูดออกเสียง (Web Speech TTS)
- **เซิร์ฟเวอร์** = กาว ไม่มี logic AI เอง แค่เรียก `ask_ai()` และเก็บ/แจกสถานะ
- **สมอง** = ไม่แตะ

**เหตุผลที่ "แยกจอ" แทบไม่เพิ่มงาน:** ทั้งใบหน้าและจอเนื้อหาต้องรู้สถานะ (ฟัง/คิด/ตอบ) จากสมองอยู่แล้ว พอมีช่อง `/state` กลางแล้ว จะแสดงผล 1 จอหรือ 2 จอก็เรื่องเล็ก

## ส่วนประกอบ (แต่ละตัวมีหน้าที่เดียว)

### `web/server.py`
- **หน้าที่:** เสิร์ฟหน้าเว็บ + เป็นสะพานระหว่างเบราว์เซอร์กับสมอง + เก็บสถานะกลาง
- **endpoint:**
  - `GET /` → redirect ไป `/screen`
  - `GET /screen` → ส่ง `screen.html`
  - `GET /face` → ส่ง `face.html`
  - `POST /ask` รับ `{"question": "..."}` → ตั้งสถานะ `คิด` → เรียก `ask_ai(question, college_data)` → ตั้งสถานะ `ตอบ` → คืน `{"answer": "..."}`
  - `GET /state` → คืน `{"state": "<สถานะปัจจุบัน>"}` (ให้หน้าใบหน้า poll)
  - `POST /state` รับ `{"state": "..."}` → อัปเดตสถานะกลาง (ให้หน้าเนื้อหาแจ้ง `ฟัง`/`รอ`)
- **สถานะ:** เก็บเป็นตัวแปร module-level ตัวเดียว (`รอ`/`ฟัง`/`คิด`/`ตอบ`/`ผิดพลาด`) รัน Flask แบบ threaded เพื่อให้ `/state` ตอบได้ระหว่าง `/ask` กำลังทำงาน
- **โหลดฐานข้อมูล:** เรียก `load_college_data()` ครั้งเดียวตอนสตาร์ท

### `web/static/screen.*` (จอเนื้อหา — ตัวคุมงาน)
- ปุ่มไมค์ใหญ่ "กดเพื่อพูด" → `SpeechRecognition` (lang = `th-TH`) → ได้ข้อความ
- ช่องพิมพ์ + ปุ่มส่ง → ทางสำรองเมื่อไม่อยากพูด
- เมื่อเริ่มฟัง: `POST /state {"state":"ฟัง"}`
- ส่งคำถาม: `POST /ask` → ได้คำตอบ → เพิ่มบับเบิลผู้ใช้/หุ่นในบทสนทนา
- พูดคำตอบ: `SpeechSynthesis` (lang = `th-TH`) เมื่อพูดจบ `POST /state {"state":"รอ"}`
- โชว์โลโก้/ภาพต้อนรับตอนสถานะ `รอ`

### `web/static/face.*` (จอใบหน้า — แสดงผลล้วน)
- วาดตา 2 ข้าง + ปาก (CSS/SVG)
- poll `GET /state` ทุก ~200ms → เปลี่ยนการแสดงออกตามตาราง "ใบหน้า ↔ สถานะ"
- ไม่ยุ่งกับไมค์/ลำโพง/สมอง

## ลูปการทำงาน (data flow)

```
1. รอ      → ใบหน้ากระพริบตาช้าๆ / จอเนื้อหาโชว์โลโก้ต้อนรับ
2. ผู้ใช้กดไมค์ → screen ตั้งสถานะ "ฟัง" → ใบหน้าตาเบิ่ง
              → เบราว์เซอร์ฟังไทย → ได้ข้อความคำถาม
              (หรือพิมพ์ในช่อง → ข้ามมาขั้น 3)
3. screen POST /ask → server ตั้งสถานะ "คิด" → ใบหน้าตากลอกขึ้น
              → server เรียก ask_ai() → ได้คำตอบ
4. server ตั้งสถานะ "ตอบ" → ใบหน้าปากขยับ/ยิ้ม
              → screen โชว์บับเบิลคำตอบ + เบราว์เซอร์พูดไทย
5. พูดจบ   → screen ตั้งสถานะ "รอ" → วนกลับขั้น 1
```

## ใบหน้า ↔ 5 สถานะ (ใช้สถานะเดิมจาก `output.py`)

| สถานะ | ใบหน้า | ไฟ LED จำลอง (บนจอ) |
|-------|--------|---------------------|
| รอ | กระพริบตาช้าๆ | ขาว |
| ฟัง | ตาเบิ่ง สดใส | ฟ้า |
| คิด | ตากลอกขึ้น | เหลือง |
| ตอบ | ยิ้ม/ปากขยับ | เขียว |
| ผิดพลาด | หน้างง | แดง |

ไฟ LED จำลองด้วยแถบสีบนจอเนื้อหา (ของจริงไว้เฟสฮาร์ดแวร์)

## เสียง (Web Speech API)

- **ฟัง (STT):** `SpeechRecognition` / `webkitSpeechRecognition`, `lang = "th-TH"` — รองรับดีใน Chrome/Edge (ใช้เอนจิน Google เบื้องหลัง ต้องต่อเน็ต) และต้องเปิดผ่าน `https` หรือ `localhost` (เปิดบนคอมผ่าน `localhost` ใช้ได้)
- **พูด (TTS):** `speechSynthesis`, เลือกเสียง `th-TH` ถ้ามีในเครื่อง (Windows มักมีเสียงไทย) ถ้าไม่มีให้พูดด้วยเสียงเริ่มต้น
- **ค่า config:** `SPEECH_LANG = "th-TH"`

---

## การกันพลาด (error handling)

- **ฟังไม่ออก / ไม่ได้ยิน** → ขึ้นข้อความ "ไม่ได้ยินค่ะ ลองพิมพ์ดูไหมคะ" + ชี้ไปช่องพิมพ์ (นี่คือ fallback ตามที่ต้องการ) แล้วกลับสถานะ `รอ`
- **เบราว์เซอร์ไม่รองรับ Web Speech** (เช่น Firefox) → ตรวจตอนโหลด ถ้าไม่มี `SpeechRecognition` ให้ซ่อนปุ่มไมค์ ใช้ช่องพิมพ์แทนอัตโนมัติ
- **AI / เน็ตล่ม** (`ask_ai` โยน exception) → server คืน error, screen ตั้งสถานะ `ผิดพลาด` + ข้อความ "ระบบขัดข้อง ลองใหม่อีกครั้ง"
- **คำถามว่าง** → ใช้ข้อความ `_NO_QUESTION` เดิมใน `ai.py`
- **กันตอบมั่ว (หัวใจ)** → ใช้ system prompt เดิมใน `ai.py` ไม่เปลี่ยน

## การทดสอบ (บนคอม ไม่ต้องมีอุปกรณ์)

1. `pip install -r requirements.txt` (เพิ่ม flask) แล้วรัน `web/server.py`
2. เปิด **2 หน้าต่างเบราว์เซอร์** (Chrome/Edge): `localhost:<port>/face` และ `localhost:<port>/screen` วางคู่กัน
3. **ชุดเทสต์กันตอบมั่ว (สำคัญสุด):**
   - ถามเรื่อง*มี*ข้อมูล (สาขาที่เปิดสอน, เบอร์โทร, ที่อยู่) → ต้องตอบถูกตามฐานข้อมูล
   - ถามเรื่อง*ไม่มี*ข้อมูล (หอพัก, ค่าเทอมตัวเลขเป๊ะ, สระว่ายน้ำ) → ต้องตอบ "ยังไม่มีข้อมูล + ติดต่อเจ้าหน้าที่" ไม่แต่ง
4. ทดสอบไมค์จริงผ่านไมค์โน้ตบุ๊กใน Chrome + ทดสอบช่องพิมพ์สำรอง
5. เช็กว่าใบหน้าเปลี่ยนตามสถานะจริง (กดไมค์ → ตาเบิ่ง, ระหว่างคิด → ตากลอก, ตอบ → ปากขยับ)

> หมายเหตุ Windows: รัน Python ด้วย full path + ตั้ง UTF-8 ตามที่บันทึกไว้ใน memory `python-env-windows`

## ความปลอดภัย

- API key (Gemini/Claude) อยู่ฝั่งเซิร์ฟเวอร์ใน `.env` เท่านั้น **ไม่ส่งไปเบราว์เซอร์** — หน้าเว็บคุยกับ Flask ของเราเอง ไม่เรียก AI ตรงจาก JS
- `.gitignore` เดิมครอบ `.env` อยู่แล้ว
- เซิร์ฟเวอร์ผูกกับ `127.0.0.1` (เครื่องตัวเอง) เป็นค่าเริ่มต้น

## ขอบเขตที่ "ยังไม่ทำตอนนี้" (ไว้เฟสประกอบกล่อง)

- ตัวกล่อง/โครงสร้างหุ่นจริง
- แยกขึ้น 2 จอกายภาพจริง (หัว + ลำตัว)
- ไฟ LED วงแหวนจริง (rpi_ws281x)
- ลำโพง/ไมค์แยกของกล่อง
- ทางเลือกอัปเกรดเป็น Google Cloud STT/TTS หากเสียงเบราว์เซอร์ไม่พอบนฮาร์ดแวร์จริง
- ตั้งเปิดเว็บเต็มจออัตโนมัติ (kiosk) ตอนหุ่นบูต

## เฟสถัดไป

1. **(เวอร์ชันนี้)** เว็บแอปกล่องพูดได้ ทดสอบบนคอม — พูด/พิมพ์ → ตอบเสียง + จอ + ใบหน้า
2. ประกอบกล่อง + จอสัมผัส + Raspberry Pi → เปิดหน้าเว็บเดิมขึ้น 2 จอจริง
3. ต่อไฟ LED จริง + ปรับเสียง/ไมค์ฮาร์ดแวร์ให้ชัวร์
4. เก็บ log คำถาม-คำตอบ ทำสถิติตอนแข่ง/เล่มจบ
